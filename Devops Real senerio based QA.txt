Difference Between NSG and ASG
“In Azure networking, Network Security Groups (NSGs) and Application Security Groups (ASGs) work together to secure network traffic, but they serve different purposes:

	1. Network Security Group (NSG):
		•	What it is: A security filter that contains a list of Access Control List (ACL) rules to allow or deny inbound/outbound network traffic.
		•	Scope: Can be applied to a Subnet or an individual Network Interface (NIC) of a VM.
		•	How it works: Filters traffic based on 5-tuple information (Source IP, Source Port, Destination IP, Destination Port, Protocol).

	2. Application Security Group (ASG):
		•	What it is: An object that allows you to group virtual machines based on their application function (e.g., web-servers, app-servers, db-servers).
		•	Scope: Used as a source or destination in NSG rules. You do not apply rules to an ASG directly; instead, you write an NSG rule and use the ASG as the source or destination.
		•	How it works: Simplifies NSG rule management by letting you write a single rule for a group of VMs rather than writing rules for individual IP addresses.

Summary of Differences:
| Feature | NSG (Network Security Group) | ASG (Application Security Group) |
| :--- | :--- | :--- |
| Primary Purpose | Filters network traffic at the packet level. | Groups VMs logically to simplify security rules. |
| Configuration | Contains allow/deny security rules. | Contains no rules; acts as a label/grouping. |
| Association | Associated with Subnets or NICs. | Associated with VM Network Interfaces (NICs). |
| Key Benefit | Acts as a firewall for subnets/VMs. | Eliminates the need to maintain IP addresses in NSG rules. |

Example Scenario:
Instead of writing three NSG rules to allow traffic from Web-VM1, Web-VM2, and Web-VM3 to DB-VM1 on port 3306, we associate the Web VMs with `web-asg` and the DB VM with `db-asg`. Then, we write a single NSG rule: Allow inbound from `web-asg` to `db-asg` on port 3306.”

How Do You Secure Secrets?
“We follow a strict, multi-layered approach to secure secrets (passwords, API keys, certificates) across our environments:

	1. Centralized Secret Storage:
		•	We use Azure Key Vault (or AWS Secrets Manager) as the single source of truth for all secrets. No secrets are ever hardcoded in the application code, configuration files, or committed to Git.
	2. Integration with AKS (Secrets Store CSI Driver):
		•	Instead of storing secrets as Kubernetes secret objects in plaintext, we use the Azure Key Vault Secrets Store CSI Driver. This mounts the secrets directly from Key Vault into the application pods as temporary volumes (in-memory tmpfs) or syncs them as Kubernetes secrets.
	3. Access Control (RBAC & Managed Identities):
		•	We use Azure AD Workload Identity (or IAM Roles for Service Accounts in AWS) to assign a managed identity to our AKS Pods. The Pods authenticate to Key Vault securely without requiring any credentials.
		•	We enforce the principle of least privilege using Key Vault Access Policies or Azure RBAC, granting only read-only access to specific secrets.
	4. Pipeline Security:
		•	In Azure DevOps / Jenkins, we use Service Connections and secret variables. Secret masking is enabled to prevent secrets from being printed in the pipeline console logs.”

Difference Between git fetch and git pull?
“In Git, both `git fetch` and `git pull` are used to retrieve changes from a remote repository, but they differ in how they apply those changes to your local branch:

	1. git fetch:
		•	What it does: Downloads all new data (branches, commits, tags) from the remote repository to your local repository.
		•	Impact: It does not modify your working directory or your current local branch. It simply updates your remote-tracking branches (e.g., `origin/main`).
		•	Safety: 100% safe. It allows you to inspect the changes using `git log` or `git diff` before merging them.
	2. git pull:
		•	What it does: Performs a `git fetch` followed immediately by a `git merge` (or `git rebase` if configured) to merge the remote changes into your current local branch.
		•	Impact: It updates your local working files directly.
		•	Safety: Less safe. If there are conflicting changes between your local commits and the remote commits, it will immediately trigger a merge conflict that you must resolve.

Summary:
`git pull` = `git fetch` + `git merge`

Best Practice:
I prefer using `git fetch` followed by `git diff origin/main` to review what my teammates have pushed, and then running `git merge` or `git rebase` manually. This prevents accidental merge conflicts and keeps the commit history clean.”

Difference Between Deployment and StatefulSet?
“In Kubernetes, both Deployments and StatefulSets are controllers used to manage pods, but they are designed for different types of workloads:

	1. Deployment:
		•	Use Case: Designed for stateless applications (e.g., web servers, frontend apps, stateless APIs).
		•	Pod Identity: Pods are interchangeable and have random, non-sticky identities (e.g., `web-7f84-abcde`, `web-7f84-fghij`).
		•	Storage: All replicas typically share the same persistent volume (if any) or do not require persistent storage at all.
		•	Scaling: Pods are created and destroyed in parallel, in no particular order.
	2. StatefulSet:
		•	Use Case: Designed for stateful applications that require unique identities and persistent data (e.g., databases like PostgreSQL, MySQL, MongoDB, or Kafka, Elasticsearch clusters).
		•	Pod Identity: Pods have a sticky, unique network identifier and a persistent index starting from 0 (e.g., `db-0`, `db-1`, `db-2`).
		•	Storage: Each Pod gets its own dedicated Persistent Volume (PV) via a `volumeClaimTemplates`. When a pod is rescheduled, it reconnects to the exact same PV.
		•	Scaling: Pods are created, updated, and terminated sequentially (e.g., `db-1` is not created until `db-0` is running and healthy).

Summary of Differences:
| Feature | Deployment | StatefulSet |
| :--- | :--- | :--- |
| Workload Type | Stateless (Web Apps, APIs). | Stateful (Databases, Message Queues). |
| Pod Naming | Random hash suffix (e.g., `app-xyz12`). | Ordinal index (e.g., `mysql-0`, `mysql-1`). |
| Storage | Shared volume or no volume. | Dedicated volume per Pod (Volume Claim Template). |
| DNS / Network | Shared service DNS. | Headless service providing unique DNS per Pod (e.g., `mysql-0.db-service`). |
| Scaling Order | Parallel (fast scaling). | Sequential (one-by-one, ordered). |”

1. Modularize Everything (Don’t write monolithic code)
“Terraform code should always be modularized instead of writing everything in a single file.
For example:
	•	Separate module for VNet 
	•	Separate module for AKS 
	•	Separate module for Storage Account 
	•	Separate module for VM 
Benefits:
	•	Reusability 
	•	Easy maintenance 
	•	Better readability 
	•	Team collaboration becomes easier 
In our project, we maintained separate reusable modules and called them from environment-specific code.”

2. Use Remote State + Locking
“In production environments, Terraform state should never be stored locally.
We used Azure Blob Storage as remote backend storage for Terraform state files.
Benefits:
	•	Centralized state management 
	•	Team collaboration 
	•	Prevents state file loss 
	•	Better security 
We also enabled state locking to avoid multiple users modifying infrastructure simultaneously.
If state gets locked accidentally, we can release it using:
terraform force-unlock ```”  ---  # 3. What is Terraform State & why is it critical?
“Terraform state is a file that stores the current mapping between Terraform configuration and real infrastructure resources.
Whenever Terraform creates infrastructure, it stores resource details inside the state file.
The state file contains:
	•	Resource IDs 
	•	Metadata 
	•	Dependencies 
	•	Current infrastructure status 
Terraform compares:
	•	Desired configuration vs 
	•	Current state 
to determine:
	•	What to create 
	•	What to update 
	•	What to delete 
Without state, Terraform cannot properly track infrastructure.
For example: If a VM already exists but Terraform has no state information, Terraform may try to create the same VM again.
That’s why state is extremely critical in Terraform.”

Why Terraform State is Important
1. Infrastructure Tracking
Terraform knows which resources it manages.
2. Change Detection
Terraform identifies infrastructure drift and required modifications.
3. Dependency Management
Terraform maintains resource relationships and execution order.
4. Performance Optimization
Terraform does not query every resource every time.
5. Collaboration
Remote state allows teams to work together safely.

Remote State Best Practice
“In production, we never store state locally.
We store Terraform state in:
	•	Azure Blob Storage 
	•	S3 bucket 
	•	Terraform Cloud 
We also enable:
	•	State locking 
	•	Versioning 
	•	Backup 
This prevents corruption and concurrent modifications.”

6. If One Resource Needs to Be Deleted in Terraform, How Will You Do It?
“If I need to delete a specific resource managed by Terraform, I can use three primary approaches depending on the situation:

	1. The Standard/Clean Method (Recommended):
		•	Remove the resource block (or module call) from the Terraform configuration files (`.tf`).
		•	Run `terraform plan` to verify that Terraform intends to destroy only that specific resource.
		•	Run `terraform apply` to execute the deletion.
	2. The Targeted Destroy Method (For Emergency/Temporary Deletions):
		•	Run the targeted destroy command:
		  `terraform destroy -target=azurerm_virtual_machine.my_vm`
		•	Note: Use this with caution, as it leaves the resource block in your code, meaning the next time anyone runs `terraform apply`, the resource will be recreated.
	3. The State Exclusion Method (To delete from Terraform management without destroying the physical resource):
		•	Run `terraform state rm azurerm_virtual_machine.my_vm`.
		•	Delete the resource block from the `.tf` code.
		•	This removes the resource from the state file without deleting it from Azure/AWS.”

8. What is Terraform Import?
“Terraform Import is a command used to bring existing, manually created cloud resources under Terraform management without recreating them.

How it works:
	1. Write the Code: First, write the Terraform resource block in the `.tf` file for the resource to be imported.
	   Example:
	   ```hcl
	   resource "azurerm_resource_group" "rg" {
	     name     = "my-existing-rg"
	     location = "East US"
	   }
	   ```
	2. Run the Import Command: Execute the import command, mapping the Terraform resource address to the actual cloud resource ID:
	   `terraform import azurerm_resource_group.rg /subscriptions/.../resourceGroups/my-existing-rg`
	3. Update the State: Terraform queries the cloud provider, retrieves the resource attributes, and writes them directly into the `terraform.tfstate` file.
	4. Verify Alignment: Run `terraform plan` to ensure that the code matches the imported state. If there are differences, adjust the `.tf` code until `terraform plan` shows 'No changes'.”

9. Scenario: Production VM Accidentally Deleted. What Will You Do?
“If a production VM is accidentally deleted, I will follow a structured disaster recovery and incident response workflow:

	1. Immediate Mitigation:
		•	Check Monitoring & Alerts: Verify the alerts in Azure Monitor/Grafana to confirm when it went down and its impact.
		•	Assess the State: Check if the VM was managed by Terraform. If yes, inspect the Terraform state file.
	2. Reconstruction:
		•	Recreate via Terraform: Since we practice Infrastructure as Code, I will run `terraform plan` to see the delta. Since the physical VM is missing, Terraform will detect the difference and plan to recreate it. Then I will run `terraform apply` to recreate the VM with the exact same configuration (CPU, Memory, Network, IP, Security Groups).
		•	Restore Data from Backup: If the VM has stateful data, I will restore the latest disk snapshot or backup from Azure Backup / AWS Backup.
	3. Root Cause Analysis (RCA):
		•	Audit Logs: Inspect Azure Activity Logs or AWS CloudTrail to identify who or what deleted the VM (human error, script, or automated process).
	4. Prevention for Future:
		•	Enable Resource Locks: Apply `CanNotDelete` locks on all production resource groups.
		•	Enable Soft Delete: Enable soft-delete on disks and Key Vaults.
		•	Deploy VMSS: Use Virtual Machine Scale Sets (VMSS) with multiple instances behind a Load Balancer instead of a single standalone VM.”

10. Scenario: Application is Down in Kubernetes. How Will You Troubleshoot?
“To troubleshoot an application that is down in a Kubernetes cluster, I follow a systematic bottom-up approach:

	1. Check Pod Status:
		•	Command: `kubectl get pods -n <namespace>`
		•	Look for statuses like `CrashLoopBackOff`, `Pending`, `ImagePullBackOff`, or `Error`.
	2. Inspect the Pod Details:
		•	Command: `kubectl describe pod <pod-name> -n <namespace>`
		•	Check the 'Events' section at the bottom. This reveals issues like failed scheduling (insufficient CPU/Memory), volume mount failures, or failed readiness/liveness probes.
	3. Check Application Logs:
		•	Command: `kubectl logs <pod-name> -n <namespace> --tail=100` (add `-c <container-name>` if multi-container).
		•	This helps identify application-level crashes, unhandled exceptions, database connection failures, or missing environment variables.
	4. Check Cluster & Node Status:
		•	Command: `kubectl get nodes` and `kubectl describe node <node-name>` to check if the node is OutOfDisk, MemoryPressure, or NotReady.
	5. Check Networking & Services:
		•	Command: `kubectl get svc -n <namespace>` and `kubectl describe svc <service-name>`.
		•	Verify if the service selector matches the pod labels. Test DNS resolution and network policies.”

12. Scenario: Disk Space Full on Linux Server
“If a Linux server's disk space is full, it can cause services to crash. I resolve this using the following process:

	1. Check Disk Space Usage:
		•	Run `df -h` to see which mount point (e.g., `/`, `/var`) is 100% full.
	2. Identify Large Files or Directories:
		•	Run `du -sh /* 2>/dev/null | sort -h` starting from the root directory, and recursively drill down to find the offending directories (e.g., `/var/log`, `/var/lib/docker`).
		•	To find the top 10 largest files: `find / -type f -size +100M -exec ls -lh {} + 2>/dev/null | awk '{ print $5, $9 }' | sort -hr | head -n 10`
	3. Safe Cleanup:
		•	Clear Log Files safely (do not delete them while active): Clear them using `cat /dev/null > /var/log/syslog` or `truncate -s 0 /var/log/app.log`.
		•	Docker Cleanup: If Docker is taking up space, run `docker system prune -a --volumes` to remove unused containers, networks, images, and build cache.
		•	Clean Package Manager Cache: Run `apt-get clean` (Debian/Ubuntu) or `yum clean all` (RHEL/CentOS).
	4. Long-term Prevention:
		•	Configure log rotation (`logrotate`) to compress and delete old logs.
		•	Set up disk space alerts in Prometheus/Grafana or Azure Monitor to notify the team when disk usage exceeds 80%.”

13. Scenario: How Will You Secure Terraform State?
“Terraform state files contain sensitive information (like database passwords, API keys, and private keys) in plain text. I secure it using the following best practices:

	1. Use a Remote Backend:
		•	Never store `terraform.tfstate` locally or commit it to Git.
		•	Use a secure remote backend like Azure Blob Storage, AWS S3, or HashiCorp Consul.
	2. Enable Encryption:
		•	Ensure the storage bucket/container has Encryption-at-Rest enabled (e.g., using Azure Storage Service Encryption or AWS KMS).
		•	Ensure all communication is encrypted in transit using HTTPS/TLS.
	3. Enable State Locking:
		•	Enable state locking (using Azure Blob Storage native leases or AWS DynamoDB) to prevent concurrent writes that can corrupt the state.
	4. Implement Strict RBAC:
		•	Restrict access to the storage account/S3 bucket using Role-Based Access Control (RBAC). Only the CI/CD pipeline service principal and senior administrators should have read/write access.
	5. Use Secrets Managers:
		•	Instead of hardcoding secrets in variables, fetch them dynamically at runtime from Azure Key Vault or AWS Secrets Manager so they are not hardcoded in the state files where possible.”

14. Scenario: How Will You Deploy Zero-Downtime Application?
“To deploy an application with zero downtime, I use a combination of Kubernetes deployment strategies and Terraform lifecycle configurations:

	1. Kubernetes Deployment Strategy (Rolling Update):
		•	Use the `RollingUpdate` strategy in the Deployment manifest.
		•	Configure `maxSurge` (e.g., 25%) to define how many extra pods can be created during the update.
		•	Configure `maxUnavailable` (e.g., 0%) to ensure that no existing pods are terminated until new pods are ready.
		•	Define `readinessProbe` and `livenessProbe` so Kubernetes knows exactly when the new pod is healthy and ready to receive traffic before routing users to it.
	2. Terraform Lifecycle:
		•	If deploying VMs or infrastructure behind a load balancer, use the `create_before_destroy = true` lifecycle hook in Terraform. This ensures the new resource is provisioned and healthy before the old one is destroyed.
	3. Blue-Green or Canary Deployments:
		•	For critical applications, I configure Blue-Green deployments (switching traffic between two identical environments using a Load Balancer/DNS) or Canary deployments (routing 5-10% of traffic to the new version using Istio or NGINX Ingress to test stability before full rollout).”

		  ```
		  ├── dev/
		  │   ├── main.tf
		  │   └── terraform.tfvars
		  ├── prod/
		  │   ├── main.tf
		  │   └── terraform.tfvars
		  ```
	2. Reusable Modules:
		•	Write core infrastructure (like VNet, AKS, VMs) as reusable Terraform modules. The environment-specific directories simply call these modules with different input parameters.
	3. Separate Backend State:
		•	Each environment must have its own remote state file in a separate storage container/bucket to prevent accidental modification of production state during dev testing.
	4. Variable Separation:
		•	Use `.tfvars` files to define environment-specific values (e.g., small VM sizes for Dev, large VM sizes for Prod).”

17. Scenario: Application Works Locally but Fails in Docker
“When an application works on a developer's local machine but fails inside a Docker container, it is usually due to environment differences. I troubleshoot this using these steps:

	1. Inspect Container Logs:
		•	Run `docker logs <container-id>` to see the application's stdout/stderr. This usually reveals startup crashes, missing packages, or database connection errors.
	2. Check Port Mapping:
		•	Verify that the port exposed inside the Dockerfile matches the port mapped in the run command.
		  Example: `docker run -p 8080:80 my-image` (maps host port 8080 to container port 80).
	3. Check Environment Variables:
		•	Ensure all required environment variables, config files, or secrets are passed into the container at runtime using the `-e` flag or `--env-file`.
	4. Debug Interactively:
		•	Run a shell inside the running or failed container to inspect files and test network connectivity:
		  `docker exec -it <container-id> /bin/sh` or `docker run --entrypoint /bin/sh -it my-image`.
	5. Check Base Image and Permissions:
		•	Ensure the container has the correct file permissions and is not running into permission issues when writing to directories, and that the base OS image has all required system libraries.”

18. Scenario: How Will You Monitor Production Infrastructure?
“I implement a comprehensive 3-tier monitoring strategy to ensure production reliability, visibility, and quick alerting:

	1. Infrastructure Monitoring (Metrics):
		•	Use Prometheus and Grafana or Azure Monitor / AWS CloudWatch.
		•	Monitor system resource metrics: CPU usage, Memory utilization, Disk I/O, Network bandwidth, and Node status.
	2. Application & Kubernetes Monitoring:
		•	Monitor application metrics using Prometheus exporters (like Node Exporter or JMX Exporter).
		•	For Kubernetes, monitor Pod restarts, crash loops, API server latency, and ingress controller traffic.
	3. Log Aggregation:
		•	Use the ELK Stack (Elasticsearch, Logstash, Kibana) or EFK (Fluentd) or Azure Log Analytics.
		•	All container stdout/stderr logs are forwarded to a central repository where we can query and set up alerts for error spikes.
	4. Alerting:
		•	Configure Alertmanager or Azure Alerts to send high-priority notifications to Slack, PagerDuty, or Microsoft Teams when critical thresholds are breached (e.g., Disk > 85%, Pod restarts > 5 in 10 mins).”

19. Scenario: How Will You Secure Kubernetes Cluster?
“Securing a Kubernetes cluster requires a multi-layered security approach (Defense in Depth) across the entire stack:

	1. Network Security:
		•	Deploy the cluster as a Private Cluster (control plane and worker nodes do not have public IPs).
		•	Use Network Policies to restrict pod-to-pod communication (implement a default-deny policy and only allow necessary traffic).
	2. Authentication & Authorization:
		•	Enable Role-Based Access Control (RBAC) and integrate it with Azure Active Directory (AAD) or AWS IAM.
		•	Enforce the Principle of Least Privilege: developers get read-only access, CI/CD gets deployment access.
	3. Container Security:
		•	Run containers as non-root users (configure `securityContext` in pod specs).
		•	Use read-only root filesystems where possible.
		•	Enable vulnerability scanning for Docker images in the container registry (e.g., Azure Container Registry or Trivy) before deployment.
	4. Secret Management:
		•	Do not store secrets in plaintext config files. Use Kubernetes Secrets (preferably encrypted at rest using KMS keys) or integrate with Azure Key Vault using the Secret Store CSI Driver.”

20. Scenario: How Will You Handle Secrets in CI/CD Pipeline?
“Managing secrets securely in CI/CD pipelines is critical to prevent credential leaks. I follow these strict security practices:

	1. Use Integrated Secrets Managers:
		•	Store all credentials, API keys, and certificates in a dedicated secrets manager like Azure Key Vault, HashiCorp Vault, or AWS Secrets Manager.
	2. Fetch Secrets Dynamically:
		•	Configure the CI/CD pipeline (Jenkins, Azure Pipelines, or GitHub Actions) to fetch secrets dynamically at runtime using secure service connections or OIDC, rather than hardcoding them in the repository.
	3. Masking in Logs:
		•	Ensure secret masking is enabled in the pipeline settings so that if a secret is accidentally printed to the console, it is automatically replaced with asterisks (`***`).
		•	Never store secrets in plaintext in variables files (`terraform.tfvars`) or Git. Use environment variables injected by the runner or encrypted variables (like GitHub Encrypted Secrets).
	4. Rotate Credentials:
		•	Implement regular rotation policies for passwords, SSH keys, and API tokens.”
