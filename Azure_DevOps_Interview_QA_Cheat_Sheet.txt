Azure DevOps Interview Q&A Cheat Sheet

1. Explain Azure Role-Based Access Control (RBAC)
Azure RBAC is an authorization system that controls who can access Azure resources and what actions they can perform.
RBAC consists of three components:
	•	Security Principal – User, Group, Service Principal, or Managed Identity. 
	•	Role Definition – Collection of permissions (Owner, Contributor, Reader, etc.). 
	•	Scope – Where the role is assigned. 
	•	Management Group 
	•	Subscription 
	•	Resource Group 
	•	Resource 
Example: Assigning the Contributor role to a user at the Resource Group scope allows them to manage resources in that Resource Group but not grant permissions to others.

2. Basic steps to give Contributor and Reader role
	•	Sign in to Azure Portal. 
	•	Go to the Subscription/Resource Group/Resource. 
	•	Select Access Control (IAM). 
	•	Click Add > Add Role Assignment. 
	•	Select the role: 
	•	Contributor 
	•	Reader 
	•	Select the User, Group, or Service Principal. 
	•	Click Review + Assign.

3. How would you migrate a 3-tier application from on-premises to Azure with minimal downtime?
A typical approach:
Step 1: Assessment
	•	Inventory servers and applications. 
	•	Identify dependencies. 
	•	Assess databases and storage. 
Step 2: Azure Landing Zone
	•	Create Resource Groups. 
	•	Configure VNets and Subnets. 
	•	Configure NSGs. 
	•	Set up Azure Load Balancer or Application Gateway. 
Step 3: Migrate Each Tier
	•	Web Tier → Azure VM or AKS. 
	•	Application Tier → Azure VM Scale Set or AKS. 
	•	Database Tier → Azure SQL Database or Azure Database for PostgreSQL/MySQL (or SQL VM if required). 
Step 4: Data Synchronization
	•	Use Azure Database Migration Service. 
	•	Synchronize data until cutover. 
Step 5: Validation
	•	Test application functionality. 
	•	Validate network connectivity. 
	•	Perform performance testing. 
Step 6: Cutover
	•	Schedule a maintenance window. 
	•	Update DNS. 
	•	Switch production traffic. 
	•	Monitor application health. 
This phased approach minimizes downtime and reduces migration risk.

7. Grafana shows a 404 API error. How would you troubleshoot and find the RCA?
Troubleshooting steps:
	•	Confirm the error in Grafana dashboards. 
	•	Check API logs. 
	•	Verify if the API endpoint exists. 
	•	Check Kubernetes Ingress configuration. 
	•	Verify Service and Endpoints:
kubectl get svc kubectl get endpoints
	•	Check pod health:
kubectl get pods kubectl logs
	•	Test the API directly:
curl http://api-url
	•	Check DNS resolution. 
	•	Verify Azure Application Gateway or Load Balancer routing. 
	•	Review recent deployments. 
Possible Root Causes (RCA):
	•	Incorrect API URL. 
	•	Wrong ingress path. 
	•	Service selector mismatch. 
	•	Backend pods unavailable. 
	•	Failed deployment. 
	•	DNS issues. 
	•	Application routing changes. 

8. You are getting a 404 error. How would you troubleshoot from an infrastructure perspective?
As an Infrastructure Engineer, I would check the infrastructure layer before assuming it's an application issue.
Step 1: DNS
	•	Is the domain resolving correctly?
nslookup <domain>
Step 2: Network
	•	Verify connectivity.
ping traceroute
Step 3: Load Balancer/Application Gateway
	•	Check backend health. 
	•	Verify listener and routing rules. 
Step 4: Kubernetes
	•	Check Ingress:
kubectl get ingress kubectl describe ingress
	•	Check Services:
kubectl get svc
	•	Check Pods:
kubectl get pods kubectl logs <pod-name>
Step 5: Web Server
	•	Verify NGINX or Apache configuration. 
	•	Check reverse proxy rules. 
Step 6: Firewall/NSG
	•	Ensure required ports (80/443) are open. 
Step 7: SSL
	•	Validate certificates if HTTPS is used. 
Step 8: Deployment
	•	Confirm the latest deployment is healthy. 
	•	Roll back if needed. 
Conclusion: If all infrastructure components (DNS, networking, load balancer, ingress, services, and pods) are healthy, I would escalate the issue to the application team with my findings, as the 404 is likely due to an application routing or API configuration issue rather than the infrastructure itself.

1. Trying to clone a repository and getting a 403 error. What will you do?
A 403 Forbidden error means authentication succeeded or reached the server, but you don't have permission to access the repository.

	•	Verify the repository URL. 
	•	Check if I have access to the repository. 
	•	Verify my Git credentials or Personal Access Token (PAT).

	•	Check if SSH keys are configured correctly (if using SSH). 
	•	Confirm I'm logged into the correct GitHub/Azure DevOps account. 
	•	Test network/proxy settings if behind a corporate firewall. 
	•	Ask the repository administrator to verify my permissions. 

2. How to reduce Docker image size if it's 2.5 GB?
Ways to reduce image size:
	•	Use lightweight base images (e.g., alpine or slim images). 
	•	Use multi-stage builds. 
	•	Remove unnecessary packages and cache:
apt-get clean rm -rf /var/lib/apt/lists/*
	•	Combine RUN commands to reduce image layers. 
	•	Add a .dockerignore file to exclude unnecessary files. 
	•	Copy only required files instead of the entire project. 
	•	Avoid installing development tools in the production image. 
This can often reduce an image from several GB to a few hundred MB.

	•	Subscription 
	•	Resource Group 
	•	Parameters 
	•	Customize the policy (e.g., allowed regions). 
Common Policy Effects:
	•	Deny – Blocks non-compliant deployments. 
	•	Audit – Reports non-compliance. 
	•	Append – Adds settings during deployment. 
	•	Modify – Updates resources automatically. 
	•	DeployIfNotExists – Deploys required resources if missing. 
	•	AuditIfNotExists – Checks whether required resources exist. 
Example:
	•	Policy: Allow only Standard VM SKUs. 
	•	Assigned at the Production Management Group. 
	•	Any non-compliant VM deployment is denied. 

6. Explain Hub-and-Spoke Network Architecture.
Answer:
Hub-and-Spoke is a network topology where a central Hub VNet provides shared services, and multiple Spoke VNets host application workloads.
Diagram:
                    On-Premises                          │                   VPN / ExpressRoute                          │                     Hub VNet         ┌──────────┼──────────┐         │          │          │  Azure Firewall  Bastion   DNS/Shared Services         │  ┌──────┴───────────────┐  │                      │ Spoke VNet 1       Spoke VNet 2 (Web App)          (Database)  │                      │ AKS/VMs              SQL/VMs
Hub VNet contains:
	•	Azure Firewall 
	•	VPN Gateway 
	•	ExpressRoute Gateway 
	•	Azure Bastion 
	•	DNS 
	•	Shared services 
Spoke VNets contain:
	•	Application workloads 
	•	AKS clusters 
	•	Virtual Machines 
	•	Databases 
Communication:
	•	Spokes connect to the Hub using VNet Peering. 
	•	Traffic between spokes typically passes through the Hub for centralized inspection and control. 
Benefits:
	•	Centralized security 
	•	Reduced management overhead 
	•	Scalable architecture 
	•	Better isolation between applications 
	•	Cost optimization by sharing common services 
Interview Tip:
If asked why this architecture is preferred, explain that it allows organizations to centralize networking and security while keeping application environments isolated, which is ideal for enterprise-scale Azure deployments.
