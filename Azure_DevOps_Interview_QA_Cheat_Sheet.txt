1. Explain Azure Role-Based Access Control (RBAC)
Azure RBAC is an authorization system that controls who can access Azure resources and what actions they can perform.
RBAC consists of three components:
	•	Security Principal – User, Group, Service Principal, or Managed Identity. 
	•	Role Definition – Collection of permissions (Owner, Contributor, Reader, etc.). 
	•	Scope – Where the role is assigned. 
	•	Management Group 
	•	Subscription 
	•	Resource Group 
	•	Resource 
Example: Assigning the Contributor role to a user at the Resource Group scope allows them to manage resources in that Resource Group but not grant permissions to others.

	•	Click Add > Add Role Assignment. 
	•	Select the role: 
	•	Contributor 
	•	Reader 
	•	Select the User, Group, or Service Principal. 
	•	Click Review + Assign. 

3. How can I give user permission to files in Linux?
“In Linux, file permissions and ownership can be managed using a variety of commands to control access:

	1. Changing File Ownership (`chown`):
		•   To change the owner of a file: `sudo chown username filename`
		•   To change both the owner and the group: `sudo chown username:groupname filename`
		•   To apply ownership changes recursively to all files in a directory: `sudo chown -R username:groupname /path/to/directory`

	2. Changing File Permissions (`chmod`):
		•   Using Octal (Numeric) Mode:
			•   `chmod 755 filename` -> Read, write, execute (7) for Owner; read and execute (5) for Group and Others.
			•   `chmod 644 filename` -> Read and write (6) for Owner; read-only (4) for Group and Others.
		•   Using Symbolic Mode:
			•   `chmod u+x filename` -> Adds execute (+x) permission for the owner (u).
			•   `chmod g-w filename` -> Removes write (-w) permission for the group (g).

	3. Managing Group Membership:
		•   Add a user to a supplementary group to grant them the group's permissions:
		    `sudo usermod -aG groupname username`

	4. Advanced Access Control Lists (ACLs):
		•   For fine-grained permissions (e.g., granting a specific user access without changing the owner or group):
		    `setfacl -m u:username:rwx filename`
		•   To view the ACL permissions: `getfacl filename`

To verify the permissions, run `ls -l filename`, which displays the owner, group, and the read (r), write (w), and execute (x) flags.”

5. How can I give Owner permission to a user?
“To grant 'Owner' permissions to a user in Microsoft Azure, you use Role-Based Access Control (RBAC) via the Azure Portal, Azure CLI, or Terraform:

	1. Via the Azure Portal:
		•   Navigate to the target Scope (e.g., a Subscription, Resource Group, or individual Resource).
		•   Click on the **Access Control (IAM)** tab in the left-hand menu.
		•   Click **+ Add** and select **Add role assignment**.
		•   Under the **Role** tab, search for and select the **Owner** role, then click Next.
		•   Under the **Members** tab, click **+ Select members**, search for the user's name or email, select them, and click Select.
		•   Click **Review + assign** to complete the process.

	2. Via the Azure CLI:
		•   Run the following command:
		    `az role assignment create --assignee "user@example.com" --role "Owner" --scope "/subscriptions/<subscription-id>/resourceGroups/<rg-name>"`

What the Owner Role Can Do:
	•   **Full Control**: Create, read, update, and delete all resources within the assigned scope.
	•   **Access Management**: Delegate permissions to other users by creating new RBAC role assignments (this distinguishes the 'Owner' role from the 'Contributor' role, which cannot manage access control).”

12. Share 10 Linux commands
“Here are 10 essential Linux commands that I use daily for system administration, directory navigation, and troubleshooting:

	5. `rm` (Remove): Deletes files or directories.

	6. `cp` (Copy): Copies files or directories from one location to another.
	   *Example*: `cp -r /src /dest`
	7. `mv` (Move / Rename): Moves files or renames them.
	   *Example*: `mv oldname.txt newname.txt`
	8. `cat` (Concatenate): Outputs the contents of a file to the terminal.
	   *Example*: `cat /etc/resolv.conf`
	9. `chmod` (Change Mode): Modifies file and directory access permissions.
	   *Example*: `chmod 755 script.sh`
	10. `chown` (Change Owner): Modifies the owner and group of a file or directory.
	    *Example*: `chown -R admin:webgroup /var/www`”

15. If error occurs in Linux without hardware or service issues, what will you check?
“If an error occurs on a Linux server but the hardware is healthy and the service daemon is running, the issue is likely due to OS-level resource exhaustion, configuration drift, or permission blocks. I check the following:

	1. System and Kernel Logs:
		•   **System Log**: Run `journalctl -xe` or inspect `/var/log/syslog` (or `/var/log/messages` on RHEL) to check for system-level errors.
		•   **Kernel Ring Buffer**: Run `dmesg -T` to check for kernel-level alerts, such as the Out-Of-Memory (OOM) killer terminating processes.

	2. Resource Utilization:
		•   **Disk Space**: Run `df -h` to verify if any mount point is 100% full, which prevents writing logs or temp files.
		•   **Inodes**: Run `df -i` to check if the filesystem has run out of inodes (preventing new file creation even if disk space is free).
		•   **Memory & Swap**: Run `free -m` or `top` / `htop` to check if physical memory is exhausted and the system is swapping heavily.
		•   **CPU & Load Average**: Run `uptime` and `top` to check the CPU load average and identify high-CPU processes.

	3. Permissions and File Handles:
		•   **Permissions**: Verify file/directory ownership and read/write/execute permissions using `ls -la`.
		•   **Open Files**: Run `ulimit -n` and check `/proc/sys/fs/file-max` to see if the process has reached the maximum open file descriptors limit.

	4. Network and DNS:
		•   **DNS Resolution**: Run `nslookup google.com` or `dig` to verify if DNS resolution is working.
		•   **Network Ports**: Run `netstat -tuln` or `ss -tuln` to verify if the application is listening on the expected ports.”

4. Difference between Ingress Controller and API Gateway
“While both Ingress Controllers and API Gateways manage incoming traffic (North-South traffic) in a microservices architecture, they serve different purposes and operate at different levels:

	1. Ingress Controller:
		•	What it is: A Kubernetes-native resource that manages external access to services within a cluster. It acts as a smart reverse proxy.
		•	Scope: Operates primarily at the entry point of a Kubernetes cluster.
		•	Core Functions:
			•	Layer 7 HTTP/HTTPS routing based on hostnames (e.g., api.example.com) or paths (e.g., /users).
			•	SSL/TLS termination.
			•	Basic path-based load balancing.
		•	Examples: NGINX Ingress Controller, Traefik, HAProxy, Azure AGIC.

	2. API Gateway:
		•	What it is: A comprehensive API management layer that acts as a single entry point for all clients. It is application-aware and designed to manage APIs as products.
		•	Scope: Can operate inside or outside Kubernetes, managing traffic across multiple clusters, VMs, or legacy systems.
		•	Core Functions:
			•	Advanced Security: Authentication (JWT, OAuth2, API Keys) and Authorization.
			•	Traffic Control: Rate limiting, throttling, request/response transformation.
			•	API Lifecycle Management: Caching, request retries, circuit breaking, telemetry, and API analytics.
			•	Developer Portal: API documentation and usage tracking.
		•	Examples: Kong, Apigee, Azure API Management (APIM), AWS API Gateway.

Summary of Differences:
| Feature | Ingress Controller | API Gateway |
| :--- | :--- | :--- |
| Primary Focus | Simple routing and load balancing within Kubernetes. | Advanced API management, security, and traffic control. |
| Scope | Cluster-bound (runs inside Kubernetes). | Global (can span across Kubernetes, VMs, and cloud services). |
| Key Capabilities | Path/Host routing, SSL termination. | Rate limiting, JWT validation, caching, transformations. |
| Target Audience | DevOps / Platform Engineers. | API Developers / Product Owners. |

In a production enterprise architecture, we often use both together: an external API Gateway (like Azure APIM or Kong) handles authentication, rate limiting, and global routing, and then forwards the traffic to an Ingress Controller (like NGINX) inside the AKS cluster to distribute traffic to specific pods.”

5. Difference between PV and PVC in Kubernetes
“In Kubernetes, Persistent Volumes (PV) and Persistent Volume Claims (PVC) decouple storage provisioning from storage consumption:

	1. Persistent Volume (PV):
		•	What it is: The actual storage resource in the cluster, provisioned by an administrator or dynamically via a StorageClass. It represents a physical disk in the cloud (like Azure Disk, Azure Files, or AWS EBS).
		•	Lifecycle: It has its own lifecycle independent of any individual pod that uses it.
		•	Scope: It is a cluster-wide resource (not bound to a namespace).

	2. Persistent Volume Claim (PVC):
		•	What it is: A request for storage by a developer or a Pod. It specifies the size, access mode (e.g., ReadWriteOnce, ReadWriteMany), and optionally a specific StorageClass.
		•	Lifecycle: It is bound to a specific namespace. When a pod needs storage, it references the PVC, and Kubernetes automatically binds it to a matching PV.

Flow of Storage:
Pod ---> requests storage via ---> PVC ---> binds to ---> PV ---> points to physical ---> Storage (Azure Disk/S3)

Key Differences:
| Feature | Persistent Volume (PV) | Persistent Volume Claim (PVC) |
| :--- | :--- | :--- |
| Concept | The actual storage resource (like a VM). | A request for storage (like a VM request). |
| Created By | Cluster Administrator or StorageClass. | Application Developer or Helm Chart. |
| Scope | Cluster-wide (Namespace-agnostic). | Namespace-scoped. |
| Reusability | Can be reused depending on Reclaim Policy. | Bound to a single pod/workload at a time. |

Reclaim Policies:
	•	Retain: Keeps the PV and its data after PVC deletion (requires manual cleanup).
	•	Delete: Automatically deletes the PV and physical storage when the PVC is deleted.
	•	Recycle: Performs a basic scrub (`rm -rf`) and makes it available again (deprecated).”

7. Explain the working of CNI in Kubernetes
“CNI (Container Network Interface) is a standardized specification that defines how network plugins should configure networking for Kubernetes Pods.

How CNI Works Step-by-Step:
	1. Pod Creation: The API Server schedules a Pod to a Node, and the local Kubelet begins creating the Pod.
	2. Network Namespace Creation: Kubelet creates a network namespace (netns) for the Pod.
	3. CNI Trigger: Kubelet calls the CNI plugin (configured on the node) with `ADD` command, passing the container ID and netns path.
	4. IP Assignment & Routing:
		•	The CNI plugin allocates an IP address to the Pod from the node's CIDR block.
		•	It creates a virtual ethernet pair (`veth` pair): one end is attached to the pod's namespace (as `eth0`), and the other end is attached to the host network (e.g., a bridge).
	5. Communication: CNI configures the routing tables so the Pod can communicate with other Pods across the cluster.

Popular CNI Plugins:
	•	Azure CNI: Gives each Pod a real IP address from the Azure VNet (highly performant, but consumes many VNet IPs).
	•	Kubenet: A basic plugin where Pods get IPs from a private range, and the host performs NAT (saves IPs but has routing overhead).
	•	Calico: Uses BGP for high-performance routing and supports fine-grained Network Policies for security.
	•	Cilium: Uses eBPF (Extended Berkeley Packet Filter) for secure, high-performance networking and observability without iptables overhead.”

8. Why do we require multiple containers in a Pod?
“While the standard practice is to run a single container per Pod, there are scenarios where multiple containers are required to run together. These containers share the same network namespace (can communicate via `localhost`), storage volumes, and lifecycle.

Common Use Cases:
	1. Sidecar Pattern: A helper container that assists the main container without modifying its code.
	   Example: The main container runs a web app, and the sidecar container (like Fluent Bit) collects logs and sends them to Elasticsearch.
	2. Ambassador Pattern: A proxy container that manages external connections for the main container.
	   Example: A database proxy that handles connection pooling and encryption to a remote database.
	3. Adapter Pattern: A container that standardizes the output of the main container.
	   Example: Formats application-specific metrics into a format Prometheus can scrape.
	4. Init Containers: Containers that run and complete before the main application container starts.
	   Example: A container that waits for a database to become online or downloads configuration files.”

9. What is the use case of a Sidecar Container?
“A Sidecar Container runs alongside the main application container in the same Pod to extend and enhance its functionality. This separation of concerns allows the main application to remain simple and focused.

Common Use Cases:
	1. Log Shipping: The main application writes logs to a shared volume, and the sidecar container (e.g., Fluent Bit, Logstash) reads those logs and ships them to a central logging server.
	2. Monitoring & Metrics: A sidecar container collects application-level metrics (e.g., Prometheus JMX Exporter) and exposes them on a `/metrics` endpoint.
	3. Service Mesh Proxy: In a service mesh (like Istio or Linkerd), a sidecar proxy (Envoy) is injected into every Pod. It intercepts all inbound and outbound network traffic to handle service discovery, mutual TLS (mTLS), load balancing, and rate limiting.
	4. Configuration Syncing: A sidecar container monitors a remote repository (like Git or Consul) for configuration changes and automatically updates the local config files used by the main application without restarting the pod.”

10. Difference between count and for_each in Terraform
“Both `count` and `for_each` are meta-arguments used in Terraform to create multiple instances of a resource, but they behave differently and are suited for different use cases:

	1. count:
		•	How it works: Creates a list of resources based on a numeric index.
		•	Example:
		  ```hcl
		  resource "azurerm_virtual_machine" "vm" {
		    count = 3
		    name  = "vm-${count.index}"
		  }
		  ```
		•	Limitation: Since resources are tracked by index (e.g., `vm[0]`, `vm[1]`, `vm[2]`), if you delete `vm[1]`, Terraform will shift the index of `vm[2]` to `vm[1]`, causing it to destroy and recreate the remaining resources.

	2. for_each:
		•	How it works: Loops through a map or set of strings, tracking each resource by a unique key.
		•	Example:
		  ```hcl
		  resource "azurerm_virtual_machine" "vm" {
		    for_each = toset(["web", "app", "db"])
		    name     = "vm-${each.key}"
		  }
		  ```
		•	Advantage: Resources are tracked by their key (e.g., `vm["web"]`, `vm["app"]`). Deleting or modifying one item has zero impact on the others.

Summary of Differences:
| Feature | count | for_each |
| :--- | :--- | :--- |
| Data Type | Number or List. | Map or Set of strings. |
| Resource Tracking | Index-based (0, 1, 2). | Key-based (string keys). |
| Best For | Creating identical resources. | Creating resources with unique configurations. |
| Deletion Safety | Low (shifting index causes recreations). | High (deleting one key does not affect others). |”

11. What is a State File in Terraform?
“The Terraform State File (`terraform.tfstate`) is a JSON file that acts as the single source of truth for your managed infrastructure. 

Key Functions:
	1. Mapping to Real Resources: It maps the resource declarations in your `.tf` files to the actual physical resources created in your cloud provider.
	2. Metadata & Dependencies: It tracks metadata, resource dependencies, and resource attributes that are not visible in your code.
	3. Performance Optimization: By caching the state of resources, Terraform avoids querying the cloud provider APIs for every single resource during a `plan` or `apply` operation.

Best Practices for Teams:
	•	Never commit state files to Git (they contain plain-text secrets).
	•	Use a Remote Backend (like Azure Storage Account) with Encryption-at-Rest enabled.
	•	Enable State Locking (using Azure Storage native leases) to prevent concurrent runs from corrupting the state file.”

12. Explain the CI/CD workflow
“A robust CI/CD (Continuous Integration / Continuous Delivery) workflow automates the software release lifecycle from code commit to production deployment:

	1. Continuous Integration (CI) Phase:
		•	Git Push: A developer pushes code or opens a Pull Request (PR).
		•	Trigger: The CI pipeline (e.g., Azure Pipelines, Jenkins, GitHub Actions) starts automatically.
		•	Build: The application is compiled, and dependencies are installed.
		•	Unit Testing: Automated unit tests are run to catch code bugs early.
		•	Code Quality & Security: Tools like SonarQube scan the code for code smells, and tfsec/Trivy scan for vulnerabilities.
		•	Artifact Creation: If all checks pass, a build artifact (e.g., a zip file, jar file, or Docker image) is generated and pushed to a registry (like ACR or Artifactory).

	2. Continuous Delivery / Deployment (CD) Phase:
		•	Deploy to Dev/QA: The artifact is deployed automatically to lower environments.
		•	Integration & UAT Testing: Automated integration tests or manual User Acceptance Testing (UAT) is performed.
		•	Manual Approval: For production environments, a gatekeeper (release manager) reviews the release and provides manual approval.
		•	Production Release: The application is deployed to production using zero-downtime strategies (like Rolling Updates or Blue-Green deployments).”

13. You need to provision an Azure Storage Account in multiple environments. What Terraform setup would you design?
“To deploy an Azure Storage Account across multiple environments (Dev, QA, Prod) while maintaining clean, reusable, and secure code, I would design a modular directory structure using remote backends:

	1. Directory Structure:
```text
terraform/
├── modules/
│   └── storage_account/
│       ├── main.tf        # Resource definition
│       ├── variables.tf   # Module input variables
│       └── outputs.tf     # Outputs (e.g., connection strings)
└── environments/
    ├── dev/
    │   ├── main.tf        # Calls the storage module
    │   ├── variables.tf
    │   ├── terraform.tfvars # Dev-specific values
    │   └── backend.tf     # Dev remote state configuration
    └── prod/
        ├── main.tf        # Calls the storage module
        ├── variables.tf
        ├── terraform.tfvars # Prod-specific values
        └── backend.tf     # Prod remote state configuration
```

	2. Key Design Principles:
		•	Reusable Modules: The `modules/storage_account/` directory contains the generic definition of the storage account. We do not hardcode environment names or sizes here.
		•	Environment Isolation: Each environment (dev, prod) has its own directory and its own `backend.tf` file pointing to a separate storage container. This ensures that a change in Dev cannot accidentally modify or corrupt the Prod state.
		•	Parameterization: We use `terraform.tfvars` in each environment to pass specific values (e.g., LRS storage for Dev to save costs, and GRS storage for Prod for high availability).”
Approach:
	•	Create a reusable Storage Account module. 
	•	Use separate variable files (dev.tfvars, qa.tfvars, prod.tfvars). 
	•	Use separate remote state files for each environment. 
	•	Deploy through Azure DevOps pipelines.

17. What is a Template in a YAML pipeline?
“A Template in an Azure DevOps YAML pipeline is a reusable file that allows you to define common pipeline logic once and share it across multiple pipelines. This follows the DRY (Don't Repeat Yourself) principle.

Key Types of Templates:
	1. Stage Templates: Define a complete deployment stage (e.g., a Dev deployment stage).
	2. Job Templates: Define a set of jobs (e.g., a build job that runs on multiple OS platforms).
	3. Step Templates: Define a sequence of tasks (e.g., a step to install Node.js, run npm build, and publish artifacts).

Benefits of Using Templates:
	•   Reusability: Avoids duplicate YAML code across different repositories or pipelines.
	•   Security & Governance: Centralizes security scans or compliance tasks in a single template that all teams must reference.
	•   Maintainability: If a task version changes, you only need to update it in the template file rather than in hundreds of individual pipelines.”

18. Difference between Parameters and Variables in a pipeline
“In Azure DevOps YAML pipelines, both parameters and variables are used to store and pass values, but they differ significantly in their evaluation time, mutability, and security:

	1. Parameters (Compile-Time):
		•   What they are: Strongly-typed inputs defined at the top of the YAML file. They are evaluated at compile-time (before the pipeline starts running).
		•   Mutability: Immutable. Once the pipeline starts, their values cannot be changed.
		•   Use Case: Used to control the structure of the pipeline (e.g., passing a boolean to decide whether to run a deployment stage, or choosing a target environment from a dropdown list).

	2. Variables (Runtime):
		•   What they are: Dynamic values that can be defined in the YAML, in variable groups, or via the UI. They are evaluated at runtime.
		•   Mutability: Mutable. Their values can be updated dynamically during pipeline execution using logging commands (e.g., `##vso[task.setvariable]`).
		•   Use Case: Used to store secrets, build numbers, or dynamic outputs from one task to be used in a later task.

Summary of Differences:
| Feature | Parameters | Variables |
| :--- | :--- | :--- |
| Evaluation Time | Compile-Time (before execution). | Runtime (during execution). |
| Mutability | Immutable (read-only). | Mutable (can be changed). |
| Data Types | Strongly typed (string, number, boolean, object). | Only strings. |
| Security | Highly secure (cannot be injected at runtime). | Vulnerable to runtime injection if not handled carefully. |
| UI Prompting | Can prompt the user for input before running. | Cannot prompt the user with dropdowns. |”

19. How do you change a username using a Git command?
“To change the author name associated with your Git commits, you use the `git config` command. You can set it at different scopes depending on your needs:

	1. For the Current Repository Only:
		•   Run the command inside the specific repository folder:
		    `git config user.name "John Doe"`
		•   This writes the configuration to the `.git/config` file of that repository.

	2. Globally (For All Repositories on Your Machine):
		•   Run the command with the `--global` flag:
		    `git config --global user.name "John Doe"`
		•   This writes the configuration to the global Git config file (typically `~/.gitconfig` on Linux/Mac).

	3. To Change the Associated Email Address:
		•   You should also update your email, as Git commits track both name and email:
		    `git config --global user.email "john.doe@example.com"`

	4. To Verify the Configuration:
		•   List all active configurations to confirm the changes:
		    `git config --list` or `git config user.name`”
